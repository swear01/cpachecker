# Case Study: Context Unlock via Initial Precision Bootstrap

## 1. Summary

Bootstrap predicates converted a ZERO_CONTEXT_TIMEOUT run into a context-producing CEGAR run with 39 refinements and 3 context dumps. The benchmarks remains unsolved — B5-MR generated semantically correct repair predicates but failed the output contract validator (SSA-name contamination).

## 2. Benchmark

- **Name**: `up` (loop-invgen)
- **Source**: SV-COMP 2026 ReachSafety-Loops
- **Assertion**: `__VERIFIER_assert(k > 0)` at line in second loop
- **Structure**: Two sequential loops. First loop increments `i` and `k` while `i < n`. Second loop decrements `k` while `j < n`, with assertion `k > 0`.

## 3. Before Bootstrap

| Metric | Value |
|--------|-------|
| Status | ZERO_CONTEXT_TIMEOUT |
| Refinements | 0 |
| Context dumps | 0 |
| Spurious traces | none |
| Interpolants | none |
| Candidate fates | none |
| B5-MR operable | no |

CPAchecker at 300s could not complete even one CEGAR refinement. The initial abstraction was too weak to generate any spurious counterexample, so there was no trace to analyze and no interpolants to derive.

## 4. Bootstrap Predicates

Eight predicates were generated by LLM from source + assertion, validated, and injected into PredicateCPA precision:

| # | Predicate | Type | Fate |
|---|-----------|------|------|
| 1 | `(bvsge i (_ bv0 32))` | loop counter bound | ENTAILED |
| 2 | `(bvsge k (_ bv0 32))` | accumulator bound | ENTAILED |
| 3 | `(bvslt i n)` | loop condition | ABSTRACTION-CANDIDATE |
| 4 | `(= k i)` | loop body relation (k tracks i) | ENTAILED (at N30) |
| 5 | `(bvsge n (_ bv0 32))` | input bound | ABSTRACTION-CANDIDATE |
| 6 | `(bvsgt k (_ bv0 32))` | target property (k > 0) | coverage support |
| 7 | `(bvsle i n)` | loop guard | coverage support |
| 8 | `(bvsge k (_ bv1 32))` | inductive hint | coverage support |

**CPAchecker V-FATE classification**: 4 ENTAILED (blockFormula ⇒ p, used to strengthen interpolants via B1 injection). 2 ABSTRACTION-CANDIDATE (entered precision for PredicateCPA tracking). 2 predicates supported coverage.

**Key insight**: The entailed predicate `(= k i)` was critical — it expressed the loop body relation that `k` tracks `i` through the first loop. Without this, CPAchecker had no way to relate `k` to the loop state.

CEGAR now produces signal-rich context:

- 3 block formulas per abstraction state (SSA-SMT)
- 2 interpolants per refinement
- V-FATE logs showing ENTAILED / ABSTRACTION-CANDIDATE classification per predicate

## 6. B5-MR Attempt and Failure Analysis

### What the LLM Generated (correctly!)

The LLM analyzed the context and correctly identified the needed predicates:
- `(= k i)` — k equals i after first loop
- `(bvsgt k (bvsub n j))` — k > n - j, implying k > 0 when j < n
- `(bvsle j k)` — j <= k during second loop

### Why They Failed Validation

All 5 candidates were rejected with `REJECT_INTERNAL_SYMBOL`. The LLM used SSA-encoded variable names (`|main::i|`, `|main::k|`) copied from the B5 repair prompt's interpolant section, violating the output contract (source-level names only).

### What Would Have Worked

The semantically identical predicates with source-level names:
1. `(bvslt i n)`
2. `(= k i)`
3. `(bvslt j n)`
4. `(bvsgt k (bvsub n j))`
5. `(bvsle j k)`

These would have passed validation and could have been injected.

## 7. Why This Matters

Before bootstrap, CPAchecker produced zero signal — no traces, no interpolants, no evidence about where CEGAR was failing. After 8 initial predicates, the verifier could actually run CEGAR, producing all the normal feedback channels.

The bootstrap predicates did not solve the benchmark directly, but they **created the missing feedback channel** needed for repair-based predicate synthesis. The next repair round generated semantically correct predicates — it just failed the output format check.

## 8. Remaining Gap

1. **Output discipline**: The B5 repair prompt needs stronger enforcement that output must use source-level variable names. The LLM reads SSA-encoded interpolants and sometimes copies internal symbols.
2. **Semantic correctness**: The LLM's reasoning about the missing predicates (`k = i`, `k > n - j → k > 0`) was correct. The problem is purely a format/contract issue.
3. **Not yet solved**: Benchmark remains unsolved. Additional repair rounds with corrected predicate format may complete the proof.

## 9. Key Finding

> Bootstrap LLM predicates can unblock a CEGAR loop that previously produced zero usable context. The failure mode changed from no-signal to signal-rich.
